# 관계(Relationship)

## 정의

관계란 **엔터티의 인스턴스 사이의 논리적인 연관성**으로, 존재에 의한 관계 또는 행위에 의한 관계로 구성된다.

## 관계의 종류

### 존재 관계 vs 행위 관계

| 구분 | 설명 | 예시 |
|------|------|------|
| **존재 관계** | 엔터티 간 존재 자체로 인한 관계 | 사원은 부서에 속한다 |
| **행위 관계** | 엔터티 간 어떤 행위로 인한 관계 | 고객이 주문을 한다 |

## 관계 표기법

### 카디널리티 (Cardinality)

| 표기 | 의미 |
|------|------|
| 1:1 | 하나의 인스턴스가 하나에만 대응 |
| 1:M | 하나의 인스턴스가 여러 개에 대응 |
| M:M | 여러 인스턴스가 여러 개에 대응 (교차 엔터티로 해소) |

### 선택성 (Optionality)

| 구분 | 설명 |
|------|------|
| **필수(Mandatory)** | 반드시 관계가 존재해야 함 (‖) |
| **선택(Optional)** | 관계가 없을 수도 있음 (○) |

```
사원 ─────────── 부서
(M)               (1)
각 사원은 반드시 하나의 부서에 속한다
각 부서에는 0명 이상의 사원이 있다
```

## M:M 관계 해소

M:M 관계는 실제 데이터베이스 구현 시 **교차(Intersection) 엔터티**를 사용하여 1:M 관계 두 개로 분리한다.

```
Before:           After:
학생 ─── M:M ─── 과목      학생 ─── 1:M ─── 수강 ─── M:1 ─── 과목
                            (수강 = 교차 엔터티)
```

## 관계 명명 규칙

1. 관계명은 현재형으로 기술
2. 관계명은 능동형으로 표현
3. 각 관계는 두 개의 관계명 (양방향) 보유
4. 관계명은 명확하고 업무적으로 의미 있게 부여

예:
- 사원은 부서에 **소속된다**
- 부서는 사원을 **포함한다**

## 관계 체크사항

관계 정의 시 다음을 확인:

1. 두 엔터티 사이에 관심 있는 연관 규칙이 있는가?
2. 두 엔터티 사이에 정보의 조합이 발생하는가?
3. 업무 기술서, 장표에 관계 연결을 가능하게 하는 동사가 있는가?
4. 해당 관계가 데이터 모델에서 반드시 필요한가?

## 시험 포인트

- 존재 관계 vs 행위 관계 구분
- 카디널리티: 1:1, 1:M, M:M
- M:M → 교차 엔터티로 해소하는 방법
- 선택성: 필수(Mandatory) vs 선택(Optional)
- IE 표기법과 Barker 표기법 차이
