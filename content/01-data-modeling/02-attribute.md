# 속성(Attribute)

## 정의

속성이란 **업무에서 필요로 하는 인스턴스로 관리하고자 하는 의미상 더 이상 분리되지 않는 최소의 데이터 단위**이다.

> 예: 사원 엔터티의 속성 → 사번, 이름, 입사일, 급여 등

## 속성의 특징

- 반드시 해당 업무에서 **필요하고 관리하고자 하는** 정보여야 함
- 정규화 이론에 근거하여 **주식별자에 함수적 종속성**을 가짐
- 하나의 속성에는 **하나의 값**만 가짐 (원자성)
- 하나의 속성이 여러 값을 가지면 별도 엔터티로 분리

## 속성의 분류

### 특성에 따른 분류

| 분류 | 설명 | 예시 |
|------|------|------|
| **기본 속성** | 업무 분석에서 도출되는 본래의 속성 | 이름, 주소, 생년월일 |
| **설계 속성** | 업무 규칙화를 위해 새로 만들거나 변형 | 코드, 일련번호 |
| **파생 속성** | 다른 속성으로부터 계산/변환된 속성 | 나이(=생년월일 계산), 합계금액 |

> ⚠️ 파생 속성은 가능한 한 적게 사용 권장 (데이터 일관성 문제)

### 구성 방식에 따른 분류

| 분류 | 설명 |
|------|------|
| **단순 속성** | 더 이상 분해되지 않는 속성 |
| **복합 속성** | 여러 세부 속성으로 구성 (예: 주소 → 시/구/동) |
| **다중값 속성** | 여러 값을 가질 수 있는 속성 → 별도 엔터티로 분리 |

## 도메인(Domain)

각 속성이 가질 수 있는 **값의 범위**

```
사원명: 문자형, 최대 50자
나이:   숫자형, 0~150 정수
성별:   문자형, 'M' 또는 'F'
입사일: 날짜형, DATE
```

## 속성 명명 규칙

1. 업무에서 사용하는 이름 부여
2. **서술식 속성명 사용 금지** (예: "~의", "~에 대한")
3. 약어 사용 주의
4. 전체 데이터 모델에서 **유일한 이름** 권고
5. 엔터티에서 **유일성** 보장

## 시험 포인트

- 기본/설계/파생 속성 분류 기준과 예시
- 다중값 속성 → 별도 엔터티로 분리해야 함
- 속성의 원자성 (하나의 속성 = 하나의 값)
- 파생 속성 최소화 원칙
